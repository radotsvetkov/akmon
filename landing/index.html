// why akmon

Built for review,
not just generation

Akmon is designed for teams that need defensible software change history: explicit permissions, portable session evidence, deterministic checks, and provider control.

📋

Session evidence first

Every run can produce structured audit and evidence artifacts that survive beyond chat history and support review workflows.

🔒

Policy boundaries

Filesystem, shell, network, and tool actions are policy-evaluated, so sensitive operations are governed instead of implied.

⚙️

Single Rust binary

Terminal-first operation in local shells, SSH, and CI without dependency on editor plugin runtime stacks.

🔑

Provider optionality

Use local or hosted models with your own credentials and routing choices, instead of binding engineering workflows to one vendor.

💰

Cost transparency

Token usage, cache effects, and run cost estimates stay visible so teams can reason about budget and drift over time.

🔍

Change control + replayability

Review write hunks before approval, compare sessions with akmon diff, and validate deterministic behavior with replay workflows.

📈

Reliability gates

Use akmon slo verify and akmon slo trend to enforce reliability thresholds in CI instead of relying on ad hoc confidence.

🛡️

Governed rollout profiles

Move from dev to staging to prod using deterministic profile + policy-pack layering and effective-policy inspection.

// v2.x session evidence model

Operate with confidence

v2.1.0 builds on the v2.0.0 reviewable session lifecycle: run, inspect, replay, diff, verify, and evidence export — with resume, schema-validated tools, and tighter config wiring. Sessions remain content-addressed, cryptographically linked, replayable, and portable as AGEF bundles.

akmon --yes --output json --task "run tests and summarize" | tee run.json
akmon audit verify .akmon/audit/<session-id>.jsonl
akmon evidence verify .akmon/evidence/<session-id>.json
akmon slo verify .akmon/evidence/<session-id>.json --strict
akmon slo trend .akmon/evidence/<session-id>.json --baseline-dir .akmon/evidence/history --window 20 --strict

// get started

Install in seconds

GitHub Releases attach a real binary per platform. Writing to /usr/local/bin usually requires sudo on macOS — installing to ~/bin avoids that. If a download is HTML instead of a binary, the matching release asset is missing (cut a v* tag after the release workflow runs).

# Apple Silicon — M1, M2, M3, M4 (no sudo)
mkdir -p ~/bin
curl -L https://github.com/radotsvetkov/akmon/releases/latest/download/akmon-darwin-arm64 \
  -o ~/bin/akmon && chmod +x ~/bin/akmon
# If `akmon` is not found, add ~/bin to PATH (zsh):
# echo 'export PATH="$HOME/bin:$PATH"' >> ~/.zshrc && source ~/.zshrc
# Alternative: sudo curl … -o /usr/local/bin/akmon && sudo chmod +x /usr/local/bin/akmon

akmon --version # akmon 2.1.0
akmon config # run setup wizard
cd your-project
akmon --yes --task "summarize failing tests and propose minimal fixes"
mkdir -p ~/bin
curl -L https://github.com/radotsvetkov/akmon/releases/latest/download/akmon-darwin-x86_64 \
  -o ~/bin/akmon && chmod +x ~/bin/akmon

akmon --version
mkdir -p ~/bin
curl -L https://github.com/radotsvetkov/akmon/releases/latest/download/akmon-linux-x86_64 \
  -o ~/bin/akmon && chmod +x ~/bin/akmon

# Slim build — 3.4MB, no semantic indexing
cargo build --release --no-default-features

# Full build — 20MB, with semantic indexing
cargo build --release

mkdir -p ~/bin
cp target/release/akmon ~/bin/

curl -L https://github.com/radotsvetkov/akmon/releases/latest/download/akmon-darwin-arm64 \
  -o ~/bin/akmon && chmod +x ~/bin/akmon

# 3. Open any project
cd my-project
akmon chat # free, offline, private
